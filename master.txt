PHASE 1 — SECURITY AUDIT
════════════════════════════════════════
This phase is PRIORITY 1. Nothing goes to production with a security failure.
Read every relevant file line by line. Do not assume any protection exists — find it in the code.

A. SECRETS & CREDENTIALS:
   [ ] Zero hardcoded API keys, passwords, tokens, or secrets in any source file.
   [ ] Zero secrets committed to version control (.env files not in .gitignore).
   [ ] .gitignore explicitly excludes .env, .env.local, .env.production, and all variants.
   [ ] All secrets come from environment variables.
   [ ] .env.example exists and documents every required variable without real values.
   [ ] No credentials in comments, logs, error messages, or API responses.
   Search every file for: password, secret, api_key, apikey, token, private_key, access_key, connection_string. Report every instance and verify it is safe.

B. AUTHENTICATION:
   [ ] Every protected route has auth middleware applied. Verify by reading every route file.
   [ ] JWT tokens have expiration (exp claim) set. Read the token generation code.
   [ ] JWT secret is stored in environment variable, not hardcoded.
   [ ] Refresh token logic is implemented and secure (if applicable).
   [ ] Passwords are hashed with bcrypt, argon2, or equivalent. Read the hashing code.
   [ ] No plain-text passwords stored or logged anywhere.
   [ ] Login endpoint has rate limiting applied.
   [ ] Account lockout or brute-force protection exists on auth endpoints.
   [ ] Session invalidation works on logout (token blacklisted or session deleted).
   [ ] Password reset flow is secure (time-limited tokens, single-use, no enumeration).

C. AUTHORIZATION:
   [ ] Every admin route verifies the user has admin role. Read every admin route handler.
   [ ] Users cannot access other users' data (IDOR prevention). Read every endpoint that accepts a user ID.
   [ ] Role/permission checks happen on the backend, not only on the frontend.
   [ ] No endpoint trusts user-supplied IDs to determine ownership without verification.

D. INPUT VALIDATION & INJECTION:
   [ ] Every user input received by the backend is validated (type, length, format, allowed values).
   [ ] Every user input is sanitized before storage.
   [ ] No raw SQL queries with string concatenation or template literals using user input.
   [ ] ORM parameterized queries used throughout. Verify in every query.
   [ ] Rich text / HTML input is sanitized with a whitelist approach before storage and render (XSS).
   [ ] File uploads validate type (check actual content, not just extension), size limit, and filename.
   [ ] Uploaded files are not executable and not served from a location where they can be executed.
   [ ] NoSQL injection prevention if applicable.

E. TRANSPORT & HEADERS:
   [ ] HTTPS enforced — HTTP redirects to HTTPS. No mixed content.
   [ ] CORS configured for specific production domain only. No wildcard (*) in production.
   [ ] Security headers present: Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, Strict-Transport-Security.
   [ ] Cookies use HttpOnly, Secure, and SameSite=Strict or Lax flags.
   [ ] No sensitive data in URL parameters (tokens, passwords, PII).

F. RATE LIMITING & ABUSE PREVENTION:
   [ ] Rate limiting on auth endpoints (login, register, password reset).
   [ ] Rate limiting on any endpoint that sends emails or SMS.
   [ ] Rate limiting on payment endpoints.
   [ ] Rate limiting on file upload endpoints.
   [ ] API rate limiting configured globally.

G. DATA EXPOSURE:
   [ ] No endpoint returns password hashes, internal tokens, or sensitive internal fields.
   [ ] Error messages do not expose stack traces, file paths, or DB schema in production.
   [ ] Debug mode is OFF in production config.
   [ ] No verbose logging of sensitive user data (PII, payment data, passwords).
   [ ] Payment card data is never stored or logged (PCI compliance minimum).

H. DEPENDENCY SECURITY:
   [ ] Run a dependency audit (npm audit / yarn audit / equivalent). Read the output.
   [ ] Zero critical or high severity vulnerabilities in dependencies.
   [ ] No dependencies with known CVEs unpatched.
   [ ] Dependencies are pinned to specific versions in package.json.
   [ ] No abandoned or unmaintained packages handling security-sensitive operations.

PHASE 2 — DATABASE AUDIT
════════════════════════════════════════
Read every model, schema, migration, seed, and query file. Document:

A. CONNECTION & CONFIGURATION:
   [ ] DB connection string comes from environment variable. Never hardcoded.
   [ ] Connection pool is configured with appropriate min/max connections.
   [ ] Connection timeout and idle timeout are set.
   [ ] DB health check endpoint exists and works.
   [ ] DB connection errors are caught and handled gracefully (no unhandled promise rejections).
   [ ] Separate DB credentials for production vs development.

B. SCHEMA & MIGRATIONS:
   [ ] All migrations run cleanly on a fresh database with zero errors.
   [ ] No pending or unapplied migrations.
   [ ] Migration files are in version control.
   [ ] Rollback migrations exist for every migration.
   [ ] No destructive migrations (DROP TABLE, DROP COLUMN) without a data backup plan.
   [ ] Schema matches the models/ORM definitions exactly.
   [ ] No orphaned tables or columns not referenced by any model.

C. INDEXES & PERFORMANCE:
   [ ] Primary keys defined on every table.
   [ ] Foreign key constraints defined on every relation.
   [ ] Indexes on every foreign key column.
   [ ] Indexes on every column used in WHERE clauses or ORDER BY in queries.
   [ ] Indexes on slug, email, sku, and any other field used for lookups.
   [ ] No full table scans on tables expected to grow large.
   [ ] Composite indexes where queries filter on multiple columns.

D. DATA INTEGRITY:
   [ ] NOT NULL constraints on every required field.
   [ ] UNIQUE constraints on fields that must be unique (email, slug, sku).
   [ ] CHECK constraints or enum types for fields with limited valid values.
   [ ] Cascade rules defined for related data deletion (ON DELETE CASCADE or RESTRICT).
   [ ] No orphaned records possible due to missing foreign key constraints.

E. QUERIES & ORM:
   [ ] Every query uses parameterized inputs. No string interpolation with user data.
   [ ] N+1 query problems identified and resolved (use eager loading / joins).
   [ ] Pagination implemented for every endpoint returning lists. No unbounded queries.
   [ ] Transactions used for multi-step operations that must be atomic.
   [ ] DB errors handled specifically (unique constraint violation, not found, connection error).

F. DATA HYGIENE:
   [ ] No test data, seed data, or mock data in production database.
   [ ] No development-only accounts or admin accounts with weak passwords.
   [ ] Sensitive data (PII) is encrypted at rest where required.
   [ ] Backup strategy defined and tested.

PHASE 3 — BACKEND API AUDIT
════════════════════════════════════════
Open every route file, controller, middleware, and service file. Read every line.

A. ROUTE COMPLETENESS:
   [ ] Every route the frontend calls exists in the backend. Read every frontend API call, find the matching backend route.
   [ ] Every route defined in the backend is actually called by something. No dead routes.
   [ ] All routes are registered in the router. No routes defined but never mounted.
   [ ] Route paths are consistent (no mixed /api/v1 and /api, no trailing slash inconsistency).

B. HTTP CORRECTNESS:
   [ ] GET routes do not modify data.
   [ ] POST used for creation, PUT/PATCH for updates, DELETE for deletion.
   [ ] Correct HTTP status codes returned:
       201 for resource creation, 200 for success, 204 for no content,
       400 for bad request, 401 for unauthenticated, 403 for unauthorized,
       404 for not found, 409 for conflict, 422 for validation error, 500 for server error.
   [ ] No route returns 500 for a predictable user error.
   [ ] No route returns 200 for an error.

C. REQUEST VALIDATION:
   [ ] Every route with a request body has validation middleware.
   [ ] Required fields are validated as present and non-empty.
   [ ] Field types are validated (string, number, boolean, date, email, URL).
   [ ] Field lengths are validated (max length on all text fields).
   [ ] Enum fields only accept valid values.
   [ ] Validation errors return 422 with a clear list of which fields failed and why.

D. RESPONSE CONSISTENCY:
   [ ] All success responses follow the same shape.
   [ ] All error responses follow the same shape: { error: string, details?: any }.
   [ ] No endpoint returns raw DB objects with internal fields exposed.
   [ ] Pagination responses include: data array, total count, current page, total pages.
   [ ] No endpoint returns undefined or null as the root response.

E. ERROR HANDLING:
   [ ] Every async route handler is wrapped in try/catch or uses an async error handler.
   [ ] Unhandled promise rejections are caught globally.
   [ ] DB errors are caught and translated to appropriate HTTP responses.
   [ ] Third-party service errors are caught and handled.
   [ ] Generic 500 handler exists as a last resort.
   [ ] Error handler does NOT expose stack traces in production.

F. LOGGING:
   [ ] Request logging configured (method, path, status, response time).
   [ ] Error logging captures full error details server-side.
   [ ] No sensitive data (passwords, tokens, PII, payment data) in logs.
   [ ] Log levels configured (ERROR in production, not DEBUG).

G. PERFORMANCE:
   [ ] Response compression enabled (gzip/brotli).
   [ ] API responses cached where appropriate.
   [ ] Heavy operations use background jobs, not synchronous request handlers.
   [ ] No blocking synchronous file I/O in request handlers.
   [ ] Payload size limits configured (no accepting 1GB JSON bodies).
