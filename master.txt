ZERO REGRESSION RULE — ABSOLUTE, NON-NEGOTIABLE
════════════════════════════════════════
This is the most important rule in this entire audit after security. Read it completely before touching any file.

A REGRESSION is any change that removes, weakens, disables, bypasses, skips, hardens around, or works around an existing protection, behavior, or feature in order to make something else work. Regressions are NEVER acceptable, regardless of how minor they seem, how urgent the fix is, or how convenient the workaround would be.

SECURITY REGRESSIONS — ABSOLUTE PROHIBITION:
The following actions are PERMANENTLY FORBIDDEN. There are no exceptions. There is no context in which these are acceptable:
- Disabling, removing, commenting out, or weakening any authentication check.
- Disabling, removing, commenting out, or weakening any authorization check or role verification.
- Disabling, removing, commenting out, or weakening any input validation or sanitization.
- Disabling, removing, commenting out, or weakening any RLS policy in Supabase.
- Setting a Supabase table to RLS disabled to fix a query problem.
- Bypassing middleware to make a route work faster or easier.
- Hardcoding credentials, tokens, role values, or permission flags in source code to avoid a configuration problem.
- Moving logic that must stay on the backend to the frontend to simplify the implementation.
- Trusting frontend-supplied values (prices, roles, totals, user IDs, permissions) on the backend without re-validating from the database.
- Returning sensitive data from an endpoint "temporarily" to debug something.
- Adding a public route or open CORS policy "just for testing" and leaving it in.
- Removing rate limiting because it was causing friction during development.
- Suppressing or silencing security-related errors to make the application appear to work.

If a fix requires any of the above to work: the fix is wrong. Find a different solution.
If no other solution is immediately obvious: flag the problem as BLOCKED, document it in Spanish, and ask the developer before proceeding. Never unilaterally apply a security regression.

FRONTEND REGRESSIONS — ABSOLUTE PROHIBITION:
The following actions are PERMANENTLY FORBIDDEN in the frontend:
- Removing a loading state that was already implemented.
- Removing an error state or empty state that was already implemented.
- Removing form validation that was already implemented.
- Removing a protected route guard that was already implemented.
- Removing accessibility attributes (alt, aria-*, role, label) that were already present.
- Removing responsive behavior (media queries, flexible layouts) that was already working.
- Breaking a user flow that was previously functional to fix something else.
- Removing double-submission protection that was already implemented.
- Removing cache invalidation logic that was already working.
- Breaking the back button, deep links, or direct URL access that previously worked.

If a fix requires removing any frontend protection or behavior: find a different approach.
If no other approach is immediately obvious: flag as BLOCKED and ask the developer.

WHEN YOU ENCOUNTER A PROBLEM THAT SEEMS TO REQUIRE A REGRESSION:
1. Stop. Do not apply the regression.
2. Re-read the problem. Is there a way to solve it that does not require a regression?
3. Consider: can the backend be fixed instead of weakening the frontend? Can the DB schema be fixed instead of disabling RLS? Can configuration be corrected instead of hardcoding?
4. If you find an alternative: implement it and document why you chose it over the regression approach.
5. If you genuinely cannot find an alternative: write a detailed explanation in Spanish of the problem, what regression was considered, why it was rejected, and what information you need from the developer to solve it correctly. Then stop and wait.

REGRESSION DETECTION — ACTIVE MONITORING:
Before applying any fix, ask yourself:
- Does this change remove, weaken, or bypass any existing protection? → If yes: stop, find another way.
- Does this change make the frontend responsible for something the backend should control? → If yes: stop.
- Does this change trust user-supplied data that was previously validated server-side? → If yes: stop.
- Does this change disable a Supabase RLS policy or set a table to public? → If yes: stop.
- Does this change remove a UI protection (guard, validation, error state) that existed before? → If yes: stop.
- Does this change hardcode a value that was previously dynamic or configurable? → If yes: stop.

If any answer is yes: the change is a regression. Do not apply it.

NOTHING CRITICAL IN THE FRONTEND — ABSOLUTE RULE
════════════════════════════════════════
The frontend is a public, untrusted environment. Any critical logic placed there can be bypassed, modified, or exploited by anyone with browser developer tools. The following logic must NEVER exist only in the frontend:

PRICE & FINANCIAL LOGIC:
- Price calculation must happen on the backend using prices from the database.
- Discount calculation must happen on the backend.
- Tax calculation must happen on the backend.
- Order total must be calculated on the backend. Never trust the total sent from the frontend.
- The frontend may DISPLAY prices and totals, but must never be the source of truth for them.

ACCESS CONTROL:
- Role checks must happen on the backend on every request. A frontend route guard is UX, not security.
- Permission checks must happen on the backend. Hiding a button on the frontend is not access control.
- Admin status must be verified on the backend on every admin API call, not just on login.
- Ownership verification ("this order belongs to this user") must happen on the backend.

BUSINESS RULES:
- Stock availability must be verified on the backend at the moment of order creation.
- Coupon/discount code validity must be verified on the backend.
- Product availability (active/inactive) must be enforced on the backend.
- Order status transition rules must be enforced on the backend.
- Any rule that can result in money movement or data access must be enforced on the backend.

DATA VALIDATION:
- Input validation on the frontend is UX only. The backend must independently validate all inputs.
- Never skip backend validation because frontend validation exists.
- Never trust field values sent from the frontend (product ID, user ID, price, quantity, role) without re-validating against the database.

SENSITIVE DATA:
- API keys, service credentials, and secrets must never be in frontend code, even in environment variables prefixed for frontend exposure (NEXT_PUBLIC_, VITE_, etc.) — unless they are explicitly designed to be public (e.g., Supabase anon key with correct RLS).
- Business logic that reveals competitive information (margin calculations, cost prices) must never be in frontend code.
- Internal user data, admin data, or other users' data must never be fetched to the frontend even if not displayed.

AUDIT ENFORCEMENT:
When auditing the frontend, actively look for:
- Any fetch or API call that sends a price, total, or financial value that the backend then trusts.
- Any fetch or API call that sends a role, permission, or admin flag that the backend then trusts.
- Any environment variable exposed to the frontend that contains a secret or privileged credential.
- Any business rule implemented only in a frontend component with no backend equivalent.
- Any data fetched from the backend that includes fields the current user should not have access to.

If any of the above is found: flag it, document it in Spanish, and fix it by moving the logic to the backend. Never solve it by hiding the data on the frontend.
